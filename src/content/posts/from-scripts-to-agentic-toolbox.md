零散脚本解决一次问题很快，但时间一长会出现入口不统一、参数记不住、报告散落和重复实现。Agentic Toolbox 的目标不是重新发明七个领域，而是把已有最小工具放进同一套运行约定。

## 七个方向

首批工具覆盖：

- GitHub 与 AI 热点雷达
- Codex 和 Claude 配置体检
- Agent Skills 管理器
- 现场运维告警面板
- PDF 与 OCR 资料整理
- 网页健康检查
- 本地项目安全自查

每个工具都可以独立运行，同时接受统一入口调度。

## 先统一合同，再统一界面

工具箱先规定三件事：

1. 每个工具有稳定的命令名和帮助信息。
2. 成功、需查看和失败使用一致的退出状态。
3. 结果优先输出 Markdown、JSON 或纯文本报告，而不是只打印临时日志。

有了这些合同，网页面板只是另一种调用方式，不需要复制业务逻辑。

## 本地网页面板的职责

网页面板负责参数输入、文件上传、运行状态、报告历史和收藏。报告原文和错误输出保留在本地，不依赖远程数据库。

这样的结构让命令行继续适合自动化，让网页适合日常使用。两者共享同一套工具入口和测试。

## 产品化仍然需要边界

一个可运行工具箱不等于已经适合公网服务。涉及本机目录扫描、配置检查和敏感文件识别时，默认部署方式应该保持本地。公开展示只使用样例数据和脱敏报告。
